# CodeHarbor 功能补充分析（2026-03-18）

## 1. 分析范围

- 代码与文档范围：`README.md`、`docs/CONFIG_CATALOG.md`、`src/orchestrator.ts`、`src/channels/matrix-channel.ts`、`src/admin-server.ts`、`src/store/state-store.ts`。
- 目标：识别“已具备能力”与“下一阶段应补充的功能缺口”，并给出优先级实施建议。

## 2. 当前已具备的核心能力

- Matrix 单通道接入与消息路由，支持会话上下文、去重、防抖、限流、命令控制。
- AI 后端切换（`codex` / `claude`）与上下文桥接能力。
- 多模态基础：图片输入与音频转写（含大小限制、MIME 策略、失败回退）。
- Admin UI + Admin API（全局配置、房间配置、健康检查、审计、服务重启）。
- SQLite 持久化（会话、事件去重、配置版本、升级记录、升级锁）。

## 3. 建议补充的功能缺口（按优先级）

## P0（优先尽快完成）

1. 可观测性标准化（`/metrics` + 告警）
   - 缺口：运行指标存在但主要体现在命令输出，不便于统一监控系统接入。
   - 建议：新增 Prometheus 指标出口；补关键告警（失败率、队列等待、转写失败、升级失败）。
   - 验收：Grafana 可直接接入并看到核心 SLI；异常可触发告警通知。

2. 可靠性增强（可恢复任务队列）
   - 缺口：当前以实时处理为主，进程重启或依赖抖动时恢复策略有限。
   - 建议：引入持久化任务队列、失败重试策略、重启后的待处理恢复。
   - 验收：模拟重启后任务可恢复，失败任务可追踪并可重放。

3. 接入层抽象（多通道能力）
   - 缺口：当前核心实现偏 Matrix 单通道。
   - 建议：抽象统一 Channel 接口，预留 webhook/企业 IM 等入口。
   - 验收：新增任一新通道时无需改动核心编排逻辑。

## P1（中期完成）

1. 外部 API/Webhook 触发能力
   - 价值：便于对接 CI、工单系统、自动化平台。
   - 验收：可通过 API 发起任务、查询状态、回收结果。

2. 附件处理能力扩展（文档类）
   - 价值：补齐 pdf/docx/txt 等资料型输入场景。
   - 验收：文档可解析并进入统一上下文，且具备大小与类型策略。

3. 配置热更新覆盖扩展
   - 缺口：当前大量配置“重启后生效”。
   - 建议：将高频配置改为运行时动态生效（在保证安全前提下）。
   - 验收：关键配置可无重启生效并带审计记录。

## P2（后续优化）

1. 权限细粒度化（超越 admin/viewer）
   - 按 API、按房间、按动作分权，支持最小权限原则。

2. 会话历史产品化
   - 增加检索、导出、归档/保留策略可视化控制。

3. 策略自动化
   - 根据房间/用户/任务类型自动选择 backend/model，而非依赖手工命令切换。

## 4. 建议实施顺序（30/60 天）

- 第 1-2 周：完成 P0-1（指标与告警）+ 技术方案评审。
- 第 3-4 周：完成 P0-2（可恢复队列）最小可用版本。
- 第 5-6 周：完成 P0-3（接入层抽象）并验证扩展点。
- 第 7-8 周：推进 P1（API/Webhook + 文档附件 + 热更新增强）。
- 第 9-10 周：按业务优先级选做 P2。

## 5. 结论

CodeHarbor 当前主链路已经可用，下一阶段重点应从“功能可用”转向“可运维、可扩展、可集成”。优先完成 P0 能显著提升稳定性与工程化水平，为后续功能扩展打好底座。
