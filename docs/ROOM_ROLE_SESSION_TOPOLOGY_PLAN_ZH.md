# 房间-账号-会话-角色拓扑规划（P1/P2/P3）

## 1. 当前模型（v0.1.x）

### 1.1 核心关系
- 机器人账号（Bot Account）：由 `matrixUserId` 唯一标识，是唯一执行主体。
- 房间（Room）：通过 `conversationId` 标识；房间级配置决定是否启用、触发策略、默认工作目录。
- 会话（Session）：当前实现按 `sessionKey = channel:conversationId:senderId` 组织。
- 角色（Role）：`planner / executor / reviewer` 是同一机器人账号在同一会话中的逻辑角色，不是独立 Matrix 账号。

### 1.2 现状结论
- 目前是“单机器人账号 + 多逻辑角色 + 每用户独立会话”。
- 同一个房间里，不同发言用户会形成不同 `sessionKey`，避免上下文串线。

## 2. 管理后台配置建议（当前可执行）

### 2.1 房间配置建议
- 将需要自动响应的房间显式配置为 `enabled=true`。
- 触发策略建议按场景配置：
  - 协作群：开启 `mention` + `reply`，按需开启 `activeWindow`。
  - 公共群：建议只开 `mention`（必要时加 `prefix`），降低误触发。
- 为每个工程房间绑定稳定 `workdir`，避免跨项目执行。

### 2.2 会话与权限建议
- 管理指令（如升级/后端切换）建议只在 DM 或管理员白名单使用。
- 业务讨论在群内进行，重操作通过触发策略收敛。

## 3. 分阶段路线

## P1（本次已开始实现）
目标：先消除“看不清关系”的问题。
- 在 `/status` 里回显拓扑：`bot / room / user / session`。
- 回显触发模式与房间策略来源（default/room）与策略细项。
- 明确角色链路属于“同一会话内的逻辑角色”。

## P2（下一阶段）
目标：支持“一个房间多个机器人账号固定分工”。
- 新增 Bot Profile（账号、后端、模型、技能包、可见角色）。
- Room 绑定 Bot Profile 列表，并定义默认主答 bot。
- 引入房间级调度规则（谁可发起、谁可审批、谁可执行）。

## P3（远期）
目标：支持房间内多 bot 协同讨论与收敛。
- 支持“提案-辩论-裁决”模式。
- 支持发言轮次上限、超时、共识阈值与最终写入者锁。
- 产出结构化决策记录（proposal / objections / verdict / final patch）。

## 4. 风险控制（多 bot 前必须具备）
- 循环防护：同一 topic 的最大轮次、最大 token、最大时长。
- 写锁控制：同一工作目录同一时刻仅允许一个 bot 写入。
- 审批闸门：高风险操作（发布、迁移、删除）必须人工确认。
- 可追溯性：每次协作必须保留 requestId、botId、sessionKey、最终责任 bot。

## 5. 落地顺序建议
1. 先完成 P1 状态可视化与后台文案对齐。
2. 再做 P2 的 Bot Profile 数据模型与后台配置页。
3. 最后做 P3 协同协议与防环控制。
